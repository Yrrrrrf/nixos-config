## 0. Executive Summary

Track 3 transforms `home/modules/packages/` from a **flat, taxonomically-broken, duplication-ridden** four-file collection into a **modular, vertically-sliced, dendritic-ready** tree. Two independent axes of modularity emerge: (a) **package-data files** organized by domain (`cli.nix`, `desktop.nix`, `libs.nix`, `dev/packages.nix`) where each exports a named-attribute set the profile composes via `++`, and (b) **language modules** (`dev/lang/*.nix`) where each language is a self-contained Home-Manager module unifying Helix LSP config + LSP packages + toolchain packages — toggleable by commenting one import line. Alongside the structural lift, Track 3 executes the full classic→Rust CLI tool migration (`gitui` over `lazygit`, `skim` over `fzf`, `bottom`/`procs`/`dust`/`xh`/`delta`/`ouch`/`hyperfine`/`tokei`/`bandwhich`/`hexyl`/`tealdeer`/`gping`/`choose`/`broot`/`bacon`/`mprocs`/`jaq`/`git-cliff`), drops obsolete classics (`neofetch`, `tree`, `btop`, `fzf`, `lazygit`, `jq`), enforces the **alias-co-location invariant** (every alias lives in the same module that installs its target binary), and audits library lists for redundancy. The architecture's long-term bet: **language toggleability** (commenting `./python.nix` removes editor config + LSPs + toolchain in one rebuild) and **dendritic-readiness** (each `dev/lang/*.nix` is already a top-level-module shape, ready for promotion when flake-parts auto-import lands).

## 2. Architecture Overview

### 2.1 Two-axis modularity

```
                     ┌─────────────────────────────────────────┐
                     │       AXIS A: Package Data Files        │
                     │       (cli, desktop, libs, dev pkgs)    │
                     │                                         │
                     │   Pure data: { nav, view, text, ... }   │
                     │   Composed by profile via ++ chains     │
                     │   NOT Home Manager modules              │
                     └─────────────────────────────────────────┘
                                       │
                                       │  composed in
                                       ▼
                     ┌─────────────────────────────────────────┐
                     │           profiles/dev.nix              │
                     │  imports = [ ./default.nix              │
                     │              packages/default.nix ]     │
                     │  home.packages = libs.gui ++ cli.nav ++ │
                     │     ... ++ dev.build ++ dev.ides;       │
                     └─────────────────────────────────────────┘
                                       │
                                       │  imports
                                       ▼
                     ┌─────────────────────────────────────────┐
                     │       AXIS B: Language Modules          │
                     │       (dev/lang/*.nix)                  │
                     │                                         │
                     │   Each is a Home Manager module         │
                     │   Contains: helix LSP config + packages │
                     │   Toggled by import in lang/default.nix │
                     │   ALIAS CO-LOCATION applies here too    │
                     └─────────────────────────────────────────┘
```

### 2.2 Why two axes (not one unified shape)

A unified shape (every file is an HM module that contributes `home.packages` directly) was considered and rejected in alignment round 5. Reasons:
- Package data files need to be **composed selectively** by profile (`minimal` skips `desktop.creative`, `dev.ides`, etc.). A list of packages exposed as `cli.nav` is trivially compositional. An HM module that always sets `home.packages` is not — once imported, it always installs.
- Language files have the **opposite** requirement: they bring multiple coordinated artifacts (helix config + LSP packages + toolchain). Splitting these across "config goes here, packages go there" reintroduces the very fragmentation Track 3 eliminates. HM module shape unifies them.

The two axes are not a compromise — they reflect two different problems: **selective composition** (axis A) vs. **coherent feature toggling** (axis B).

## 3. Design Patterns & Code Standards

### 3.1 Pattern: Named Attribute Sets for Categorized Data Files

**Where applied**: `cli.nix`, `desktop.nix`, `libs.nix`, `dev/packages.nix`.

**What it is**: Each file is a function returning an attribute set whose values are package lists, named by category. The file is **pure data**, not an HM module — it does not set `home.packages`, has no `imports`, has no `programs.*` config. The profile imports it explicitly via `let cli = import ./cli.nix { inherit pkgs; }; in ...` and composes its categories with `++`.

**What problem it solves**:
- **Selective composition** — `dev` profile takes everything; `minimal` takes only `cli.nav`, `cli.view`, `cli.text`, `cli.git`. Without named attributes, every profile would have to maintain its own copy of the package list. With named attributes, the profile is a 5-line composition.
- **Cognitive locality** — when you ask "where's `bottom`?", the answer is `cli.system` (system inspection tools). The taxonomy does the searching for you.
- **No accidental drag-in** — importing `cli.nix` doesn't install anything. You must explicitly compose it. Contrast with HM-module shape where `imports = [ ./cli.nix ]` always installs the full package list.

**Standards to enforce**:
- Each file returns exactly one attribute set; no nested attribute sets except inside `dev/packages.nix` if a `lang.*` shape ever returned (it doesn't in Track 3 — `dev/lang/` replaces it entirely).
- Category names are **lowercase, single-word**, describe the *use case* not the *technology* (e.g., `system` not `monitoring`, `text` not `regex`).
- Each category list is sorted by length-of-list increasing (short categories first) to make adding new tools mechanical: append to the right list, no reordering needed.
- Comments above each list state the **classics being replaced** so the reasoning stays visible (e.g., `# bat: cat replacement`).
- No package appears in two categories within the same file. Cross-file duplication is allowed only when a package genuinely serves two purposes AND removing it from one would silently break (e.g., `bandwhich` is plausibly net + system — pick net, comment why).

**Why at year 3**: New maintainer (or future-you who forgot) can grep a tool by category in 1 second. Adding a tool is one line in one file. No "where does this go?" debate.
**Why at year 5**: When dendritic comes, these stay as data files (or get folded into category-named top-level modules). Either way, the attribute names already describe the dendritic module names — refactor is mechanical.
**Why at year 10**: The taxonomy outlives the specific tools. `cli.system` will still mean "system inspection" even if `bottom` gets replaced by something newer. The categories are stable; the contents are fluid.

### 3.2 Pattern: Vertical-Slice Home Manager Modules for Languages

**Where applied**: `dev/lang/*.nix`. Each language is one file containing helix language config + helix language-server config + `home.packages` for LSPs and toolchain.

**What it is**: A standard Home Manager module. Imports trigger `programs.helix.languages.language = [...];` set, `programs.helix.languages.language-server.<name> = {...};` set, and `home.packages = with pkgs; [...]` extended. Removing the import line removes all three contributions atomically.

**What problem it solves**:
- **Coherent feature toggling** — to drop Python, you currently have to: (1) edit `editor/helix/langs/python.nix` (or delete it), (2) edit `editor/helix/langs/default.nix` (remove the import), (3) edit `home/modules/packages/development.nix` (remove `uv` from `lang.python`), (4) edit `dev.nix` (remove `lang.python` from the `++` chain). Four edits across three files for a *single language*. After Track 3: comment one import in `dev/lang/default.nix`. Done.
- **No leakage** — the helix-config-without-toolchain bug becomes structurally impossible. If the file is imported, all three contributions land. If not, none do.
- **Dendritic readiness** — each file is already a valid top-level module (matches the dendritic shape: `imports = []` not required at top level, `pkgs` available, all module options accessible). Promotion to `modules/python.nix` post-dendritic is `mv` plus a flake.nix import line.

**Standards to enforce**:
- File name = lowercased language name (`rust.nix`, `c-based.nix`, `hyprlang.nix`).
- Module structure (always in this order):
  1. `programs.helix.languages.language = [{ name = "X"; ... }];` block (omitted only if no Helix grammar/highlight applies — `iot.nix` is the sole exception).
  2. `programs.helix.languages.language-server.<lsp-name> = { ... };` block (one per LSP).
  3. `home.packages = with pkgs; [ ... ];` (LSPs + toolchain + tools).
  4. **If a CLI tool from this language has a shell alias**, alias declared via `programs.zsh.shellAliases.<alias> = "<binary>";` at the bottom of the file. (Currently: none of the language files require aliases. Reserved for future.)
- File contains **no other language's config**. No multi-language files.
- File contains **no profile-switching logic**. The presence of the import in `dev/lang/default.nix` IS the toggle.

**Why at year 3**: Dropping a language is trivial. Adding a new language is "copy the closest existing file, edit the names". The pattern is self-perpetuating.
**Why at year 5**: When new LSPs emerge (rust-analyzer replaced, ty becomes the standard Python type-checker), the change is local to one file.
**Why at year 10**: Helix may be replaced by a different editor. The migration path is: rename `programs.helix.*` lines to the new editor's equivalent inside each file. Toolchain packages and `home.packages` lines stay. Each file is a self-contained porting unit.

### 3.3 Pattern: Alias Co-Location (universal)

**Where applied**: Everywhere a binary is installed. The alias for that binary lives in the same module as the binary itself.

**What it is**: A discipline, not a code structure. The rule: **if you remove the package, the alias removes itself in the same commit**. In practice, this means:
- Aliases for tools in `cli.nix` categories DO NOT live there (because `cli.nix` is data, not an HM module). Instead, **per-tool wrapper modules in `home/modules/shell/`** declare both the alias and reference the tool — this is how tools that need aliases (`bat`, `eza`, `gitui`, `bottom`, `skim`, `yazi`, `fd`, `ripgrep`) get set up. See section 4 for the full list.
- Aliases for language tooling go inside the language file (`dev/lang/<name>.nix`).
- Aliases for system-level tools (`lazydocker`) go inside the system module that installs them (`system/podman.nix`).
- `zsh.nix` retains zero `shellAliases` entries.

**What problem it solves**:
- **Eliminates orphan aliases** — the `lg = "lazygit"` alias surviving lazygit's removal is structurally impossible.
- **Localizes change** — replacing `bottom` with a hypothetical successor is one file change, not two.
- **Visible coupling** — reading `gitui.nix` tells you "this also defines `lg`". The coupling is documentation.

**Standards to enforce**:
- One alias per line, alphabetized within each module.
- Alias keys are the *classic* command name (`grep`, `cat`, `find`, `top`, `htop`); values are the new command. Exception: muscle-memory bridges where the user's existing typing is the modern tool's *predecessor* (`fzf=sk`, `lg=gitui` since "lg" is its own muscle-memory shortcut).
- No alias points to a binary the same module doesn't install or transitively guarantee.
- Aliases for tools from `cli.nix` categories are introduced via a **per-tool wrapper module** (see §4.2), not by polluting the data file with HM-module shape.

**Why at year 3**: Refactors don't break aliases. Aliases don't break refactors.
**Why at year 5**: When the modern tool of 2030 replaces `bottom`, the aliasing change is one file — same file as the package change.
**Why at year 10**: The pattern itself is an architectural fitness function. Any future audit ("are there orphan aliases?") is `grep -L "shellAliases" $(rg -l <binary>)` — a one-liner.

### 3.4 Pattern: Sub-Phase Atomic Commits

**Where applied**: The implementation plan (3A → 3E).

**What it is**: Each sub-phase produces exactly one git commit (or a tight commit chain inside the sub-phase) that:
- Is independently revertable (`git revert <sha>` produces a working system).
- Passes `nix flake check` cleanly.
- Passes `sudo nixos-rebuild switch` cleanly.
- Includes a per-sub-phase smoke-check list documented in section 8.

**What problem it solves**:
- **Recovery surface** — bad audit removal? `git revert` 3E. Don't lose 3C's new tools.
- **Bisection cost** — when a regression appears six months later, `git bisect` lands on the exact sub-phase, not "the big Track 3 commit".
- **Reviewability** — each sub-phase is a single concern. 3A is "no semantic changes". 3B is "drop these 6 packages and their aliases". 3C is "add these 20 packages". 3D is "split libs". 3E is "audit removals, one revertable commit per removal."

**Standards to enforce**:
- Commit message format: `track3/3<X>: <one-line summary>` (e.g., `track3/3A: lift packages/ into modular tree`).
- Sub-phase 3A introduces the new tree but **changes no installed package set** — every `which <tool>` post-3A returns the same path or a `/nix/store/*` path with identical contents (nix store-hashed, but the symlink is identical).
- Sub-phase 3E commits one library removal per commit, allowing bisect to identify the exact lib that broke a future workflow.

**Why at year 3+**: Future archeology of "why was X removed?" or "when did Y break?" is fast.

## 5. Trade-off Analysis

### 5.1 Data files vs. unified HM modules

```
DECISION: Should cli.nix / desktop.nix / libs.nix be data files or HM modules?
OPTIONS CONSIDERED:
  A. Data files exposing { nav, view, ... }; profile composes via ++.
     Pros: selective composition; no accidental drag-in; explicit profile control.
     Cons: profile files have ++ chains; importing isn't "just works".
  B. HM modules each setting home.packages directly via imports.
     Pros: single import, no composition logic in profile.
     Cons: importing always installs everything; profile loses selective control;
           minimal-vs-dev divergence harder.
  C. Hybrid — modules with a "categories enabled" option (mkOption).
     Pros: declarative profile-side selection.
     Cons: significant boilerplate per file; mkOption per category; over-engineered.
CHOSEN: A
REASON: Profile composition is the use case the user explicitly cares about
        (minimal vs dev). Option B trades that away for marginal import-line
        savings. Option C reinvents the categorization the named attributes
        already do, with NixOS option-system overhead.
REVISIT IF: Profile-switching becomes complex (5+ profiles); per-category
            options would help. Or if dendritic adoption changes the import model.
```

### 5.2 Aliases as data vs. per-tool wrapper modules

```
DECISION: Where do aliases for tools defined in the data files live?
OPTIONS CONSIDERED:
  A. Centralized in zsh.nix (status quo before Track 3).
  B. Per-tool wrapper modules in shell/tools/ (the chosen shape).
  C. Add a "shellAliases" attribute to each cli.nix category, profile reads it.
CHOSEN: B
REASON: Aligns with R8 (alias co-location universal). Zsh.nix kept clean.
        Removing a tool's alias is removing/editing one file.
        Option C bleeds zsh-specific concerns into category data.
        Option A is what we're explicitly leaving behind.
REVISIT IF: shell/tools/ grows past ~15 files; consider grouping or
            switching to a module-options-based approach.
```

### 5.3 Single dev/lang/ directory vs. split toolchain-vs-formats

```
DECISION: File-format languages (markdown, yaml, etc.) — own subdir, or mixed
          with toolchain languages?
OPTIONS CONSIDERED:
  A. Mixed in dev/lang/ alongside rust/python/etc.
  B. Split: dev/lang/toolchain/ and dev/lang/format/.
  C. Format languages stay in editor/helix/langs/, toolchain langs move out.
CHOSEN: A
REASON: One toggle file is the goal. Two toggle files (toolchain default.nix
        + format default.nix) reintroduces fragmentation. Categorical purity
        loses to operational simplicity.
        Option C breaks the unification the entire track is built on.
REVISIT IF: Format-language list grows past ~10; subdivision starts paying
            for itself.
```

### 5.4 Library audit timing — fold into 3A vs. dedicated 3E

```
DECISION: When does the full library audit (transitive removals) happen?
OPTIONS CONSIDERED:
  A. Fold into 3A (structural lift commits also include audit removals).
  B. Dedicated 3E sub-phase, isolated commits per removal.
  C. Defer entirely, post-Track 3.
CHOSEN: B
REASON: 3A's "no semantic changes" claim is preserved. Audit risk is
        isolated and bisectable. Each removal is one commit, easy to revert
        if it breaks a workflow that surfaces months later.
        Option A bundles structural and semantic changes — review-hostile.
        Option C leaves known bloat in place indefinitely.
REVISIT IF: Audit reveals nothing actually removable; sub-phase becomes a
            no-op. Then merge 3D's libs split with 3E and skip the
            "audit phase" framing.
```

### 5.5 Vulkan timing

```
DECISION: Create system/vulkan.nix and libs/vulkan.nix in Track 3?
OPTIONS CONSIDERED:
  A. Yes — minimal system/vulkan.nix, libs.nix gains vulkan attribute.
  B. No — vulkan deferred, stub-comment in nvidia.nix only.
  C. Halfway — keep vulkanLibs in libs.nix (no system module).
CHOSEN: B
REASON: User has no Vulkan workload today (round 5). Creating modules for
        non-existent workloads is speculative scaffolding.
        Stub-comment ensures the future-you doesn't forget the pattern.
REVISIT IF: Steam/Proton, Blender GPU rendering, or a Vulkan-using app
            is added.
```

### 5.6 lazydocker location

```
DECISION: Where does lazydocker live?
OPTIONS CONSIDERED:
  A. Keep in cli.nix.misc (alongside other CLI tools).
  B. Move to system/podman.nix (alias also there).
  C. Drop entirely.
CHOSEN: B
REASON: Alias co-location universal (R8). Lazydocker without podman is
        a useless binary; coupling the install to the podman.enable flag
        is correct. The mkIf block already exists.
        Option A leaves orphan-tool risk if podman is ever disabled.
        Option C: no Rust replacement exists; user uses it.
REVISIT IF: A Rust replacement for lazydocker emerges (mental note: yearly
            check).
```
